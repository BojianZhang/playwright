# post-auth-ready 边界检查

对应目录：
- `D:\playwright\shared-post-auth-ready`

这个文档只做一件事：
**把第五阶段该做什么、不该做什么、和第四阶段怎么交接、和最终 runner 怎么分层，先钉死。**

---

# 一、第五阶段是什么

第五阶段是：
- `post-auth-ready`

它的职责不是再去填写资料，也不是回头处理验证码。
它的职责是：

1. 接住第四阶段已经提交完成后的页面
2. 判断页面是否已经进入已登录态 / 已注册态可用上下文
3. 判断 session / storage / 用户态信号是否已经稳定到可交付
4. 收口主链最终成功 / 失败 / 未知结果

---

# 二、第五阶段负责什么

第五阶段负责：
- post-auth-ready ready 判断
- 已登录页面 / 工作台 / 控制台 / 首页恢复态判断
- 页面内用户态是否已经建立的确认
- session / storage 是否达到可交付状态的站内判断
- 成功/失败/未知结果确认
- 第五阶段失败分类

---

# 三、第五阶段不负责什么

第五阶段不负责：
- 首页打开
- 登录入口切换
- credential submit
- verification submit
- birthday / profile completion
- 重新拉码
- 回头修正前四阶段输入错误
- browser/context 创建
- runner 层代理切换、外层重试、结果落盘
- 账号系统对外同步、外部 API 写入、消息通知

---

# 四、第四阶段和第五阶段怎么切

## 第四阶段终点
第四阶段的成功定义是：
- 资料补全已提交
- 已确认推进到 `post-auth-ready`

第四阶段不应该继续做：
- session 最终稳定确认
- dashboard 最终 ready
- registration-complete 判定

## 第五阶段起点
第五阶段的起点是：
- 页面已经进入 `post-auth-ready` 所在上下文
- 或页面已开始呈现用户态建立后的强/弱信号

第五阶段第一步应该是：
- 确认 post-auth-ready 是否成立
- 再确认最终成功/失败/未知是否已经可收敛

---

# 五、什么算越界

以下行为如果放在第五阶段内部，通常不算越界：
- 读取 session/localStorage/cookie 的存在性摘要
- 观察用户头像、用户菜单、控制台首页、工作台、欢迎区、登出按钮等已登录信号
- 观察 URL/path 是否已经进入登录后区域
- 判断注册主链是否可以宣布完成

以下行为如果放在第五阶段内部，就属于越界或高风险：
- 重新填写邮箱/密码/验证码/生日
- 回头做前四阶段逻辑
- 修改代理、替换浏览器、重建 page/context
- 写入外部系统
- 直接做 runner 全局决策

---

# 六、第五阶段成功定义

第五阶段成功不是“所有业务逻辑都跑完”，而是：

- 已确认页面进入已登录 / 已注册后的可用上下文
- 已确认用户态 / session 信号达到可交付最低标准
- 当前注册主链可以被外层认定为 `registration-complete`

---

# 七、一句话总结

第五阶段只做一件关键事：
**把“已经从注册流程里走出来”这件事确认干净，并把最终成功/失败/未知结果交给外层。**
