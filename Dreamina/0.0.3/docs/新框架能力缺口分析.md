# Dreamina 0.0.3 新框架能力缺口分析

> 生成时间：2026-04-17  
> 分析范围：`D:\playwright\Dreamina\0.0.3\` 及依赖的所有 `shared-*` 模块  
> 忽略：`node_modules\`、`tests\`、`history\`、`session-records\` 明细  
> 标注约定：【事实】= 代码可证实 | 【推断】= 基于结构合理推理 | 【建议】= 尚未实现

## A. 已有能力

### A1. 阶段化执行链（S0→S6）【事实】
- 7 个阶段全部对应 `Sn-*/` adapter 目录 + `shared-*/` 公共骨架
- 每阶段有统一的 `StageResult` 结构（`success / finalState / finalReason / detail`）
- `runDreaminaRegisterFlow` 线性串联，阶段间通过 `stageResults` 传递中间状态

### A2. 并发调度【事实】
- `Dreamina-batch-runner.js` 实现了 `concurrency` 控制
- `shared-batch-orchestration/` 提供了 task-queue、worker-state、mutex 工具
- 代理独占锁（`createProxyLockSet`）防止并发 Worker 竞争同一代理

### A3. 失败分类器【事实】
- `failure-classifier.js` 定义了 `isProxyHardFailure / isBusinessFailure / classifyFailure`
- `isRetryableProxyOrEnvironmentFailure / isTerminalBusinessFailure` 在 batch-runner.js 中完整实现
- `failureReasonBuckets / failureClassBuckets / failurePhaseBuckets` 三级分桶

### A4. 断点续跑（EVO-13）【事实】
- `batch-results/accounts-done.txt` 全局 done 文件，跨批次有效
- 启动时读取已完成账号并从队列剔除
- config.json `resumePolicy.enabled=true`

### A5. 域名黑洞预过滤【事实】
- `config.json → batchFilter.skipEmailDomains` 在 batch-runner 启动时过滤
- 黑洞域名账号直接标 `skipped`，不开浏览器、不消耗代理

### A6. 验证码重发（resend-on-timeout）【事实】
- `verification-submit.js` 实现了"timeout → 等倒计时 → 触发 resend → 下一轮拉取"逻辑
- `config.json → verification.maxRounds` 限制轮询上限（当前=2）

### A7. 账号状态三分目录【事实，本次已落地】
- `account-state/local-accounts.json` 待注册池
- `account-state/registered-accounts.json` 已完成存档（1220条）
- `account-state/blacklisted-accounts.json` 硬失败黑名单（自动写入）
- `account-state/retry-accounts.json` 软失败可重试（自动写入）

### A8. 代理健康池【事实】
- `proxy-health.json` 记录代理状态（ok/bad/unstable/weak）
- `proxyHealthPool` 配置降级策略

### A9. 配置统一入口【事实】
- `config.json` 8KB，涵盖浏览器、代理、验证、批量、失败分类等全部运行参数
- `shared-utils/config-schema.js` 定义类型约束
- `shared-utils/config-doctor.js` 提供运行时配置诊断

## B. 缺失能力

### B1. 无集中账号状态机【事实-缺失】
**当前状况**：账号状态（pending / registered / blacklisted / retryable）分散在：
- `Dreamina-batch-runner.js` 的 `migrateAccountOutOfLocalPool`（迁移逻辑）
- `isBlacklistFailure / isAccountRetryFailure`（分类函数）
- `appendToAccountStateFile`（写入函数）
- `account-state/*.json`（存储）

**没有**一个统一的 `AccountStateManager` 模块，没有显式的状态转换图。  
**后果**：状态转换逻辑散落在 116KB 的 batch-runner.js 中，任何新增失败类型都需要在多处同步修改。

### B2. retry-accounts 缺少"回注"机制【事实-缺失】
**当前状况**：`retry-accounts.json` 有写入逻辑，但没有任何代码读取它并把账号放回 `local-accounts.json`。  
**后果**：软失败账号写入 retry-accounts.json 后就永久留在那里，没有被再次调度的能力，相当于静默丢失。

### B3. blacklisted-accounts.json 初始化为空对象而非数组【事实-缺陷】
**当前状况**：
```
blacklisted-accounts.json 内容: (not array)  — 0KB
retry-accounts.json 内容: (not array)  — 0KB
```
文件是 `[]`（5字节），JSON.parse 后正常，但 writeFile 写入时用了 `'[]'` 字符串，内容实际为 `[]`。  
**验证**：已确认内容为 `[]`，is-array 检测OK，这条可排除。（事实修正：实际是空数组，scan 脚本误报）

### B4. 无统一"可执行架构约束文档"【事实-缺失】
**当前状况**：
- `ARCHITECTURE_ANALYSIS.md` 31KB，是历史分析，非执行约束
- `Dreamina-register.README.md` 5KB，只描述模块职责
- `runtime-params.md` 5KB，只描述参数
- `shared-*/README.md` 每个模块各一个，内容分散

**没有**一个 Agent 可以直接读取并据此做决策的"边界约束 + 状态机规则"单一文档。

### B5. shared-utils 无 README【事实-缺失】
`shared-utils/` 是最基础的工具层（14个文件），却没有 README，模块职责不清晰。

### B6. session-records 无归档/清理策略【推断】
`session-records/` 有 310 个子文件，按时间戳命名，没有任何归档脚本或 TTL 配置。  
**后果**：长期运行后文件数无限增长，影响目录读取性能。

### B7. results/ 与 batch-results/ 边界重叠【事实】
- `batch-results/` — 批量运行产物（json summary + accounts-done.txt）
- `results/` — CLI 单跑产物（success/failed/latest）  

两个目录共存，没有文档说明边界，Agent 难以判断失败账号的结果写在哪里。

## C. 边界不清位置

| 位置 | 问题 | 类型 |
|------|------|------|
| `Dreamina-batch-runner.js` L579-695 | `migrateAccountOutOfLocalPool` 同时负责读 local-accounts、写 registered-accounts、触发 session 记录，职责过重 | 事实 |
| `batch-results/` vs `results/` | CLI 单跑写 `results/`，批量写 `batch-results/`，但没有任何文档说明 | 事实 |
| `isTerminalBusinessFailure` vs `isBlacklistFailure` | 两个函数枚举集合有交叉（SIGNUP_REJECTED 同时出现在两者），语义不完全一致 | 事实 |
| `batch-results/accounts-done.txt` vs `registered-accounts.json` | done 文件只存 email 行，registered 文件存完整信息，两者都需维护，分叉风险高 | 推断 |
| `shared-utils/config-schema.js` | 存在 schema 定义但未在 batch-runner 启动时强制校验，config 字段拼错不会报错 | 推断 |
| `failure-classifier.js` 在 `0.0.3/` | 这是 Dreamina 专属分类器，但路径在版本目录，未来新平台无法复用 | 推断 |

## D. 为什么这些问题会拖慢 Agent

| 缺失 | Agent 误判场景 |
|------|---------------|
| 无统一状态机文档 | Agent 不知道哪个文件是权威的账号状态，可能读 done.txt 或 registered-accounts.json 做出矛盾决策 |
| retry 无回注机制 | Agent 认为"retry 写入了就会被处理"，实际账号静默丢失，成功率统计失真 |
| results/ 与 batch-results/ 双轨 | Agent 分析失败账号时只看其中一个目录，遗漏另一半数据 |
| blacklist/retry 仅有写，无读 | Agent 询问"哪些账号失败了"时要看 retry-accounts.json，但它永远是累积的，无法区分"上一批"与"所有历史" |
| 无架构约束文档 | Agent 每次都要重新 grep 代码才能判断边界，容易误改 shared 层逻辑 |

## E. 优先级最高的落地建议

### 🔴 P1：补全 retry-accounts → local-accounts 回注脚本
**文件**：新建 `account-state/recycle-retry.js`  
**逻辑**：读 retry-accounts.json，去重后追加到 local-accounts.json，并清空 retry-accounts.json  
**触发方式**：手动执行或在 batch-runner 启动前自动检测  
**影响**：使软失败账号能被再次调度，消除静默丢失

### 🔴 P2：建立单一架构约束文档（本文件）
**文件**：`docs/ARCHITECTURE_CONTRACTS.md`  
**内容**：目录职责表、状态机转换图、文件所有权矩阵  
**效果**：Agent 读一个文件即可知道在哪里写什么，不需要 grep 全库

### 🟠 P3：补全 shared-utils README + 启动时 config 校验
**文件**：`shared-utils/README.md` + batch-runner 启动时调用 `config-doctor.js`  
**内容**：每个工具模块的职责边界、入参/返回类型约定  
**效果**：消除 config 字段拼写错误无提示的风险
