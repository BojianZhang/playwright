## 检查点 1：自动验收闭环

### 事实
- `tests/` 目录存在 6 个测试文件：`dreamina-login.spec.ts`、`example.spec.ts`、`test-1~4.spec.ts`
- 所有测试文件均为 **UI 交互式测试**（Playwright browser test），依赖真实网络和账号
- `package.json` 有 `test` 脚本，指向 `playwright test`
- **没有**任何测试文件覆盖以下内容：
  - `account-state/` 状态文件读写的正确性
  - `recycle-retry.js` 回注逻辑
  - `failure-classifier.js` 的分类准确性
  - `isBlacklistFailure / isAccountRetryFailure` 的分类覆盖率
  - batch-runner 启动时域名过滤、EVO-13 断点续跑的正确性

### 结论
**【事实-缺失】无覆盖批量注册框架核心逻辑的自动化验收测试**。  
现有测试全部是 UI E2E，并非框架逻辑单元测试。任何对 `failure-classifier.js`、`batch-runner.js`、`verification-submit.js` 的修改均无自动化安全网。

## 检查点 2：机器可读状态机和失败策略

### 事实
- 不存在任何 `state-machine.json` 或 `state-transitions.json` 文件（全路径均返回 `False`）
- 账号状态转换逻辑散落在：
  - `Dreamina-batch-runner.js` → `isBlacklistFailure()`（L~1110）
  - `Dreamina-batch-runner.js` → `isAccountRetryFailure()`（L~1120）
  - `Dreamina-batch-runner.js` → `migrateAccountOutOfLocalPool()`（L579-695）
  - `failure-classifier.js` → `isBusinessFailure / isProxyHardFailure`
- `docs/ARCHITECTURE_CONTRACTS.md` 中有 ASCII 状态机图，但这是**人类可读 Markdown**，不是机器可读 JSON/代码

### 推断
- Agent 当前判断"一条失败记录应进 blacklist 还是 retry"时，必须读 116KB 的 batch-runner.js 并定位两个函数，没有任何可直接 require() 的单一真值来源
- 失败策略（原因码 → 处置行为）与批量调度逻辑耦合在同一文件，变更一处需要手动检查另一处同步性

### 结论
**【事实-缺失】无机器可读的账号状态机定义和失败路由策略文件**。  
`failure-classifier.js` 只处理"代理失败分类"，不覆盖"账号去向决策"（blacklist/retry/registered）。

## 检查点 3：唯一编排入口

### 事实
- 当前有 **3 个编排入口**：
  
  | 文件 | 大小 | 职责 |
  |------|------|------|
  | `Dreamina/Dreamina-batch-runner.js` | 1KB | 转发层（`require('./0.0.3/...')`） |
  | `Dreamina/0.0.3/Dreamina-batch-runner.js` | **116KB** | 真正的批量编排入口（2332行） |
  | `Dreamina/0.0.3/Dreamina-register.js` | **103KB** | CLI 单账号注册入口（2532行） |

- `package.json` 的 `start / run / batch` 均指向转发层（正确）
- **但存在歧义**：`Dreamina-register.js` 单跑模式和 `Dreamina-batch-runner.js` 批量模式是**两个独立代码路径**，它们的 runtime 构建方式不同（批量通过 `createBatchRunContext`，单跑通过 `runDreaminaRegisterCli`）
- 上轮修复已同步了 `skipCredentialExistsPrecheckAfterEmail`，但单跑路径仍有其他批量特性未同步（如 `maxRounds`、`verificationResendWaitMs` 等依赖 config-reader，而不是固定字面量）

### 推断
- 单跑模式的 runtime 参数以**字面量**方式写在 `runDreaminaRegisterCli()`（L2396-2448），而批量模式从 `config.json` 读取。两者的参数演进会逐渐漂移，且无任何保护检测

### 结论
**【推断-风险】编排入口在功能上是双轨的**：单跑 CLI runtime 与 batch runtime 参数来源不同，未来容易漂移。不是"入口不唯一"，而是"同一配置有两套实现路径"。

## 检查点 4：结果/状态目录边界

### 事实
- 当前存在 **4 个结果/状态写入目录**：

  | 目录 | 写入者 | 职责 |
  |------|--------|------|
  | `account-state/` | batch-runner（成功/失败后） | **账号生命周期状态**（长期） |
  | `batch-results/` | batch-runner（每次运行） | **批量运行快照**（运行产物） |
  | `results/` | Dreamina-register.js CLI | **CLI 单跑输出**（运行产物） |
  | `session-records/` | batch-runner（注册成功时） | **Session 证据**（运行产物） |

- 逻辑上边界清晰，但：
  - `batch-results/latest/` 同时保存了每个账号的最新结果 JSON（`dreamina-batch-xxx-latest.json`），文件数与注册账号数量线性增长（已有 100+ 个 latest 文件）
  - `session-records/latest.txt` 和 `session-records/latest.jsonl` 是两个"滚动最新"文件，与时间戳批次文件并存
  - `batch-results/accounts-done.txt` 和 `account-state/registered-accounts.json` **都记录了已完成账号**，但一个是 email 行，一个是完整对象，两者须同时维护

### 结论
**【事实-问题】`accounts-done.txt` 与 `registered-accounts.json` 是双写的同一语义**，存在数据漂移风险（成功时两者必须同时写入，若任一失败则两者不一致）。

## 检查点 5：直接拖慢 Agent 的边界不清问题

| 问题 | Agent 受阻场景 | 严重度 |
|------|---------------|--------|
| 无机器可读失败路由规则 | 询问"这条失败应怎么处理"时要读 116KB 的 batch-runner.js | 🔴 高 |
| 单跑 CLI runtime 参数用字面量 | 修改 config.json 参数后单跑效果不同步，Agent 无法一次性改好 | 🟠 中 |
| `accounts-done.txt` 双写 | 排查"账号是否已注册"时需查两个文件，结论可能矛盾 | 🟠 中 |
| `batch-results/latest/` 文件数无界增长 | 枚举账号结果时需遍历 100+ jsonl 文件，响应变慢 | 🟡 低 |
| `recycle-retry.js` 无触发入口 | Agent 知道存在回注工具，但不知道何时应调用 | 🟡 低 |

## TOP 3 缺口与优先级

### 🔴 P1（最高）：缺少机器可读的失败路由策略文件
**当前状态**：失败路由逻辑（原因码 → blacklist/retry/discard）散落在 batch-runner.js 两个函数中  
**目标**：新建 `account-state/failure-routing.json`，记录每个原因码的处置行为，供 batch-runner、Agent 和未来新平台直接 require()  
**受影响文件**：`Dreamina/0.0.3/Dreamina-batch-runner.js`、`account-state/`

### 🔴 P2（高）：单跑 CLI 与批量 runtime 参数双轨漂移
**当前状态**：`runDreaminaRegisterCli()` 中有大量字面量参数（L2396-2448），与 config.json 配置节不同步  
**目标**：单跑 CLI 的所有可配参数统一从 config.json 读取，字面量仅作 fallback 默认值  
**受影响文件**：`Dreamina/0.0.3/Dreamina-register.js`

### 🟠 P3（中）：`accounts-done.txt` 与 `registered-accounts.json` 双写语义重叠
**当前状态**：注册成功时两者均写入，但若 `migrateAccountOutOfLocalPool` 失败/异常，只有 done.txt 有记录而 registered-accounts 没有，造成数据不一致  
**目标**：明确 `accounts-done.txt` 为"轻量断点续跑专用标记"，`registered-accounts.json` 为"完整档案"，两者写入失败时均应有告警日志，并增加启动时一致性校验  
**受影响文件**：`Dreamina/0.0.3/Dreamina-batch-runner.js`
