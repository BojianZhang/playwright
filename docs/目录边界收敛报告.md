> **分析时间**：2026-04-17  
> **分析范围**：`account-state/`、`session-records/`、`results/`、`batch-results/` 四个目录  
> **标注约定**：【事实】= 代码可证实 | 【推断】= 基于结构推理 | 【建议】= 尚未实施

## 1. 四目录职责定性

| 目录 | 正确定性 | 写入者 | 有读路径 | 性质 |
|------|----------|--------|---------|------|
| `account-state/` | **状态源（State Source）** | batch-runner（migrateAccountOutOfLocalPool / appendToAccountStateFile） | 是（启动时读 registered + local） | 长期跨批次持久化 |
| `batch-results/` | **批次运行产物（Run Artifact）** | batch-runner（writeBatchAccountRecordFile / writeBatchSummaryFile） | 是（accounts-done.txt 断点续跑） | 运行快照，可清理 |
| `session-records/` | **会话证据（Session Evidence）** | batch-runner（appendFirstSessionRecord） | **否**（纯只写） | 注册成功凭证，可归档 |
| `results/` | **CLI 单跑输出（CLI Output）** | Dreamina-register.js（writeCliResultFile） | 否（纯输出） | 开发调试用，完全隔离 |

## 2. 边界重叠问题清单

### 问题 1（已修复）：`KNOWN_EXISTS_FILE` 放错目录【事实】

**原状态**：
```
batch-results/latest/dreamina-known-exists.json    ← 批次运行产物目录
batch-results/latest/dreamina-known-registered.json
```

**问题**：`KNOWN_EXISTS_FILE` 本质是"跨批次已注册账号邮箱的快速跳过缓存"，语义属于**账号状态管理**，但路径放在 `batch-results/latest/`（运行产物目录）。  
批量清理 `batch-results/` 时会误删这个文件，导致下一批启动时无法快速跳过已注册账号。

**修复**：将路径迁移到 `account-state/known-accounts.json`（本次落地）。  
同时将 `KNOWN_REGISTERED_FILE`（语义相同）合并为同一文件，消除双写。

---

### 问题 2（事实，可接受）：`accounts-done.txt` 与 `registered-accounts.json` 双写

**状态**：注册成功时同时写入：
- `batch-results/accounts-done.txt`（EVO-13 断点续跑，仅存邮箱行）
- `account-state/registered-accounts.json`（完整账号信息存档）

**影响**：两者没有原子性保证，若其中一个写入失败则数据不一致。  
**结论**：【事实-可接受】两文件用途不同（一个是快速去重集合，一个是完整档案），双写是合理的。  
**建议**：启动时通过 `state-doctor.js` 检测两者差异（已实现 `done-vs-registered` 检查）。  
**当前不做代码改动**（改动影响 EVO-13 断点续跑路径，风险中等）。

---

### 问题 3（推断，低风险）：`batch-results/` 根目录混有持久状态文件

**状态**：
```
batch-results/
├── accounts-done.txt        ← 跨批次状态（应属 account-state/）
├── failure-events.jsonl     ← 跨批次追加（运行产物但有跨批次价值）
├── run-log.txt              ← 单次批次日志
└── sessions-with-country.txt
```

`accounts-done.txt` 语义上应归 `account-state/`，但代码中已通过 `.gitignore` 保护，且 EVO-13 的路径配置较多（`batch-results/accounts-done.txt` 出现在 `config.json` + batch-runner 两处），**迁移成本中等、收益低**。  

**结论**：本次不迁移，记录为已知问题。

---

### 问题 4（事实）：`session-records/` 无读路径，无清理机制

**状态**：
- 仅在注册成功时写入（`appendFirstSessionRecord`）
- 当前 310 个文件，`latest.txt` / `latest.jsonl` 滚动覆盖
- 无任何清理脚本，无归档机制

**结论**：【事实-低风险】文件数无界增长，每批注册约追加 N×2 个文件（txt+jsonl）。  
**建议**：定期执行归档（见下方附录脚本），本次不做代码修改。

---

### 问题 5（事实）：`results/` 与 `batch-results/` 完全独立，边界清晰

**状态**：
- `results/` 只有 `Dreamina-register.js` CLI 单跑写入（4 个文件，1.6MB）
- `batch-results/` 只有 `Dreamina-batch-runner.js` 批量写入（4000 个文件，1.7GB）

**结论**：无重叠，无需收敛。

## 附录：session-records 归档建议（手动执行）

```powershell
# 将 7 天前的 session 文件移到 session-records/archive/
$archiveDir = "D:\playwright\Dreamina\0.0.3\session-records\archive"
New-Item -ItemType Directory -Force $archiveDir | Out-Null
$cutoff = (Get-Date).AddDays(-7)
Get-ChildItem "D:\playwright\Dreamina\0.0.3\session-records" -File |
  Where-Object { $_.LastWriteTime -lt $cutoff -and $_.Name -ne "latest.txt" -and $_.Name -ne "latest.jsonl" } |
  Move-Item -Destination $archiveDir
```
